# 기존 연구 대비 차별점 (논문용)

## 기존 방식 분석

### 1. Shell/Bash 스크립트 기반 (대부분의 기존 도구)

```bash
# 기존 방식 예시: U-01 root 원격접속 제한 점검
result=$(grep "PermitRootLogin" /etc/ssh/sshd_config)
if [ "$result" == "PermitRootLogin no" ]; then
    echo "양호"
else
    echo "취약"
fi
```

**한계:**
- 항목마다 if-else를 하드코딩해야 함 (72개 항목 = 72개 규칙)
- 가이드라인이 바뀌면 코드를 전부 수정해야 함
- 예외 상황(주석 처리, 설정값이 여러 줄 등)에 대응 못함
- "왜 취약한지" 설명 불가, 결과만 출력
- 조치 방법을 환경에 맞게 맞춤 제공 불가

**대표 사례:**
- GitHub catember/vulnerability-check (Bash 스크립트)
- ISMS 인증용 리눅스 서버 점검 스크립트

### 2. 기존 논문: 윈도우 서버 점검 도구 (선형주 외, 2025)

- 윈도우 서버 대상 주통기 점검 도구 개발
- PowerShell 기반 규칙 점검
- **한계**: Windows만 지원, 하드코딩된 규칙, AI 미활용, 가이드라인 변경 시 코드 수정 필요

### 3. Vul-RAG 논문 (arXiv 2024)

- 소스코드 취약점 탐지에 RAG + LLM 적용
- **우리와 다른 점**: 소스코드 대상이지 시스템 설정 점검이 아님
- 하지만 RAG를 보안 분야에 적용했다는 점에서 우리 프로젝트의 이론적 근거가 됨

### 4. 상용 도구 (Nessus, OpenVAS)

- 네트워크/시스템 취약점 스캔
- **한계**: 주통기 가이드라인 기준으로 점검하지 않음, 한국 정부 기준 미지원

## 기존 vs 우리 프로젝트 비교표

| 비교 항목 | 기존 스크립트 방식 | 기존 논문 (2025) | **우리 프로젝트** |
|----------|------------------|-----------------|-----------------|
| 판정 방식 | if-else 하드코딩 | if-else 하드코딩 | **LLM 기반 지능형 판정** |
| 규칙 관리 | 코드에 내장 | 코드에 내장 | **VDB에 분리 (코드 수정 없이 업데이트)** |
| 가이드라인 변경 대응 | 코드 전체 수정 | 코드 전체 수정 | **PDF 재적재만 하면 됨** |
| 판정 근거 설명 | X (결과만 출력) | 제한적 | **자연어로 상세 설명 자동 생성** |
| 조치 방법 | 고정된 텍스트 | 고정된 텍스트 | **환경에 맞춘 맞춤형 조치 방법** |
| 지원 OS | Linux만 또는 Windows만 | Windows만 | **Linux + Windows 동시 지원** |
| 이력 비교 | X | X | **이전 진단 결과와 자동 비교** |
| 처리 속도 | 순차 (빠름) | 순차 (빠름) | **병렬 처리 (asyncio)** |
| 예외 처리 | 패턴 미매칭 시 오류 | 제한적 | **LLM이 맥락 이해하여 유연 판정** |

## 핵심 차별점 5가지

### 차별점 1: 규칙과 코드의 분리 (가장 큰 차별점)

```
[기존] 규칙이 코드 안에 하드코딩
  - 가이드라인 변경 → 코드 수정 → 재배포 → 재테스트
  - 항목 1개 추가 = 개발자가 코드 작성해야 함

[우리] 규칙이 Vector DB에 독립 저장
  - 가이드라인 변경 → PDF 재적재 → 끝
  - 항목 추가 = PDF만 추가하면 자동 반영
  - 개발자 개입 없이 보안 담당자가 직접 가이드라인 업데이트 가능
```

**논문 포인트**: 기존 도구는 점검 규칙이 코드에 종속되어 가이드라인 갱신 시 유지보수 비용이 발생하지만, 본 시스템은 RAG를 통해 규칙과 코드를 분리함으로써 가이드라인 변경에 무관하게 동작한다.

---

### 차별점 2: LLM 기반 맥락 이해 판정

```
[기존] 단순 문자열 매칭
  grep "PermitRootLogin no" → 있으면 양호, 없으면 취약
  문제: "PermitRootLogin no  # 주석" → 양호로 오판
  문제: "#PermitRootLogin no" → 주석인데 양호로 오판
  문제: 설정 파일에 중복 항목이 있으면 → 어떤 걸 따를지 모름

[우리] LLM이 맥락을 이해
  sshd_config 전체 내용을 보고:
  - 주석 처리된 것과 실제 설정을 구분
  - 중복 설정 시 마지막 값(실제 적용값)을 인식
  - Include 지시문으로 다른 파일 참조하는 경우도 고려
```

**논문 포인트**: 정규표현식 기반 판정은 설정 파일의 주석, 중복, 상속 구조 등 복잡한 맥락을 처리하지 못하는 반면, LLM 기반 판정은 설정 파일의 의미를 이해하여 정확도가 높다.

---

### 차별점 3: 자연어 기반 자동 보고서

```
[기존 보고서]
  U-01: 취약

[우리 보고서]
  U-01: root 계정 원격 접속 제한 - 취약
  판정 근거: /etc/ssh/sshd_config 파일의 38번째 줄에서 PermitRootLogin이
            yes로 설정되어 있어, root 계정으로 SSH 원격 접속이 가능한 상태입니다.
  관련 규정: 주통기 Unix 서버 U-01 (중요도: 상)
  조치 방법:
    1. vi /etc/ssh/sshd_config
    2. PermitRootLogin yes → PermitRootLogin no 변경
    3. systemctl restart sshd
  주의사항: 조치 전 일반 사용자 계정으로 SSH 접속이 가능한지 먼저 확인하세요.
```

**논문 포인트**: 보안 비전문가도 보고서만 보고 조치 가능. 기존 도구는 "취약" 판정만 하고 조치는 담당자가 가이드라인을 별도 참조해야 했음.

---

### 차별점 4: 진단 이력 비교 (지속적 보안 관리)

```
[기존] 1회성 점검 → 결과 출력 → 끝

[우리] 점검 결과를 DB에 저장 → 재점검 시 이전 결과와 자동 비교
  ┌─────────┬──────────┬──────────┬────────┐
  │ 항목     │ 이전(3월) │ 현재(6월) │ 변화   │
  ├─────────┼──────────┼──────────┼────────┤
  │ U-01    │ 취약      │ 양호      │ 개선   │
  │ U-02    │ 양호      │ 양호      │ 유지   │
  │ U-13    │ 양호      │ 취약      │ 악화 ⚠ │
  └─────────┴──────────┴──────────┴────────┘
```

**논문 포인트**: 보안 점검은 1회가 아니라 지속적으로 수행해야 하지만, 기존 도구는 이력 관리를 지원하지 않음. 본 시스템은 진단 이력을 DB에 저장하여 보안 상태 변화를 추적하고, 악화된 항목을 즉시 식별할 수 있음.

---

### 차별점 5: 병렬 처리를 통한 대규모 점검 효율화

```
[기존] 항목 순차 처리
  119개 항목 × 수작업 = 수시간~수일

[우리] asyncio 병렬 처리
  119개 항목 × LLM 동시 호출 = ~1분
```

**논문 포인트**: asyncio 기반 비동기 병렬 처리로 LLM API 호출을 동시 수행하여, 순차 처리 대비 처리 시간을 약 90% 단축.

## 논문 작성 시 활용 가능한 구조

### 제목 (안)
- "RAG 기반 주요정보통신기반시설 시스템 취약점 자동 판정 시스템"
- "LLM과 RAG를 활용한 보안 가이드라인 기반 시스템 취약점 자동 진단 연구"

### 논문 구성 (안)

```
1. 서론
   - 사이버 보안 사고 증가 현황
   - 기존 점검 방식의 한계 (수작업, 스크립트 기반)
   - 연구 목적: RAG + LLM 기반 자동 판정 시스템

2. 관련 연구
   - 주통기 취약점 분석 평가 제도
   - 기존 자동화 도구 (스크립트 기반, 상용 도구)
   - RAG 및 LLM의 보안 분야 활용 (Vul-RAG 등)

3. 시스템 설계
   - 전체 아키텍처 (수집 → VDB → RAG → LLM → 리포트)
   - 가이드라인 청킹 및 VDB 구축 방법
   - LLM 판정 프롬프트 설계
   - 병렬 처리 구조

4. 구현
   - 수집 모듈 (Linux/Windows)
   - RAG 파이프라인
   - 판정 엔진
   - 리포트 생성

5. 실험 및 평가
   - 실험 환경 (Ubuntu, Windows 10/11)
   - 판정 정확도 (기존 스크립트 vs LLM 판정)
   - 처리 속도 비교 (순차 vs 병렬)
   - 가이드라인 변경 대응 테스트

6. 결론 및 향후 연구
```

### 실험에서 보여줄 수 있는 것

| 실험 | 비교 대상 | 기대 결과 |
|------|----------|----------|
| 판정 정확도 | 스크립트 if-else vs LLM | LLM이 엣지 케이스에서 더 정확 |
| 가이드라인 변경 대응 | 코드 수정 시간 vs PDF 재적재 시간 | 코드 수정: 수시간 vs PDF 재적재: 수분 |
| 처리 속도 | 순차 처리 vs 병렬 처리 | 병렬이 ~10배 빠름 |
| 보고서 품질 | 기존(결과만) vs LLM(상세 설명) | 비전문가 이해도 설문 |

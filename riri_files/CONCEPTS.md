# 프로젝트 핵심 개념 정리

## 1. Collector

collector는 시스템에서 정보를 가져오는 모듈입니다.

이 프로젝트에서 collector의 역할은 다음까지만입니다.

- 설정값 수집
- 명령어 원본 출력 저장
- 판정에 필요한 구조화된 데이터 생성

중요한 점:

- collector는 `양호`, `취약`을 결정하지 않습니다.
- 해석은 LLM이 합니다.

즉, collector는 `센서`, LLM은 `해석기`에 가깝습니다.

## 2. Judgment

judgment는 수집된 정보가 기준 문서에 비춰 어떤 상태인지 해석하는 단계입니다.

이 프로젝트에서는 judgment가 다음을 포함합니다.

- 결과
- 이유
- 근거 문서
- 조치 방안

## 3. Guideline

guideline은 판정의 기준이 되는 문서입니다.

예:

- 주통기
- 다른 보안 가이드라인
- 회사 내부 기준 PDF

즉, 앞으로는 특정 한 문서가 아니라 `선택 가능한 기준 집합`으로 봐야 합니다.

## 4. Multi-guideline

multi-guideline은 여러 기준 문서를 동시에 관리할 수 있는 구조를 의미합니다.

이 구조가 필요해진 이유:

- 주통기만으로는 실무 요구를 모두 커버하기 어려움
- 기관/회사마다 별도 기준이 존재
- 사용자가 웹에서 원하는 기준을 선택해 판정하고 싶음

## 5. Uploaded Guideline

uploaded guideline은 사용자가 업로드한 PDF 기준 문서를 뜻합니다.

이 문서도 기본 가이드라인과 같은 흐름으로 다뤄야 합니다.

- 파싱
- 청킹
- 임베딩
- 검색
- 판정

## 6. RAG

RAG는 `Retrieval-Augmented Generation`의 약자입니다.

쉽게 말하면:

1. 관련 문서를 먼저 찾고
2. 그 문서를 LLM에 같이 주고
3. 그 근거를 바탕으로 답하게 하는 방식입니다.

이 프로젝트에서는 `수집 결과 + 선택된 가이드라인 문서`를 함께 보고 판정합니다.

## 7. Chunk

chunk는 검색하기 좋게 나눈 문서 조각입니다.

문서를 그대로 넣지 않고 나누는 이유:

- 검색 정확도 향상
- 필요한 부분만 근거로 전달 가능
- 토큰 절약

## 8. Overlap

overlap은 chunk를 나눌 때 일부 구간을 겹치게 저장하는 방식입니다.

왜 필요한가:

- 문단 경계에서 의미가 잘리는 것을 줄이기 위해
- 자유 형식 PDF에서 문맥 손실을 줄이기 위해

하지만 구조가 명확한 문서는 무조건 overlap이 필요한 것은 아닙니다.

예:

- 주통기처럼 `W-01`, `U-01` 식으로 경계가 뚜렷한 문서는 항목 단위 청킹이 더 적합
- 회사 내부 정책서처럼 자유 형식 문서는 heading + overlap 전략이 더 적합

## 9. Metadata

metadata는 문서 본문 외에 같이 저장하는 설명 정보입니다.

이 프로젝트에서 중요한 metadata 예시:

- `guideline_id`
- `guideline_name`
- `source_type`
- `item_code`
- `os_type`
- `section_title`
- `page`

metadata가 있어야 검색 범위를 제한하고 근거를 추적할 수 있습니다.

## 10. Vector DB

vector DB는 문서를 임베딩 벡터로 저장하고, 비슷한 의미의 문서를 찾기 위한 저장소입니다.

이 프로젝트에서는 가이드라인 검색에 사용됩니다.

역할:

- 수집 결과와 비슷한 기준 문서 찾기
- 선택된 guideline 안에서 관련 근거 추출

## 11. Guideline Scope

guideline scope는 검색 대상 범위를 말합니다.

앞으로 이 프로젝트에서는 아주 중요합니다.

예:

- 주통기만 선택했으면 주통기 안에서만 검색
- 회사 PDF를 선택했으면 그 문서 안에서만 검색

이 범위 제한이 없으면 RAG 품질이 떨어질 수 있습니다.

## 12. Rate Limit

rate limit은 API를 너무 많이 호출하지 못하게 하는 제한입니다.

Gemini API 2.5 전환 시에는 다음 같은 운영 제약을 고려해야 합니다.

- 분당 호출 수
- 하루 호출 수

그래서 판정 파이프라인에는 다음이 필요합니다.

- 큐
- 재시도
- 상태 저장
- 실패 처리

## 13. Structured Result

structured result는 사람이 읽는 문장만이 아니라, 필드가 나뉜 형태로 저장된 결과입니다.

예:

```json
{
  "title": "Administrator 계정 이름 바꾸기",
  "detail": "Administrator 계정이 기본 이름 그대로 존재",
  "command": "net user Administrator"
}
```

이렇게 저장하면 좋은 점:

- 웹에 표시하기 쉬움
- LLM에 전달하기 쉬움
- CSV/JSON 재활용이 쉬움

## 14. 이 프로젝트를 한 문장으로 정리하면

> 시스템에서 사실 정보를 수집하고, 사용자가 선택한 기준 문서를 근거로 RAG + LLM이 해석하는 보안 판정 플랫폼

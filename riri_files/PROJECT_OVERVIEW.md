# 프로젝트 개요

## 현재 프로젝트 정의

이 프로젝트는 Linux/Windows 시스템에서 보안 관련 사실 정보를 수집하고, 사용자가 선택한 가이드라인 또는 업로드한 PDF 기준 문서를 근거로 RAG + LLM이 판정하는 플랫폼을 만드는 것을 목표로 합니다.

예전에는 `주통기 기반 취약점 자동 진단`에 가까웠다면, 지금은 아래와 같이 방향이 넓어졌습니다.

- 단일 가이드라인이 아닌 `다중 가이드라인 지원`
- 고정 기준만이 아닌 `사용자 업로드 PDF 지원`
- collector 판단 제거, `LLM 중심 판정 구조`
- 단발성 도구가 아닌 `웹 기반 서비스`

## 전체 파이프라인

### 1. 수집

Linux/Windows collector가 실제 시스템에서 다음 정보를 수집합니다.

- 항목 코드
- 항목 제목
- 세부 수집 결과
- 원본 명령어 출력
- 수집에 사용한 명령어

이 단계에서는 `양호`, `취약` 같은 해석을 하지 않습니다.

### 2. 가이드라인 선택

사용자는 웹에서 다음 중 하나를 선택합니다.

- 미리 등록된 기본 가이드라인
- 업로드된 회사 내부 기준서

이 선택값은 이후 RAG 검색 범위를 결정합니다.

### 3. 문서 지식화

선택된 가이드라인이나 업로드된 PDF는 다음 흐름으로 처리됩니다.

1. PDF 파싱
2. 청킹
3. 메타데이터 부착
4. 임베딩 생성
5. 벡터 DB 저장

### 4. RAG 검색

항목별 수집 결과를 질의로 사용해 선택된 가이드라인 범위 안에서 관련 기준 문서를 찾습니다.

검색 시 필요한 메타데이터 예시는 다음과 같습니다.

- `guideline_id`
- `guideline_name`
- `source_type`
- `version`
- `os_type`
- `item_code`
- `section_title`
- `page`
- `chunk_id`

### 5. LLM 판정

검색된 근거 문서와 수집 결과를 함께 LLM에 전달해 판정합니다.

예정 방향:

- `Gemini API 2.5` 사용
- 항목별 호출
- rate limit 고려
- 실패 시 재시도
- 호출 이력 및 상태 저장

### 6. 결과 저장 및 보고서 생성

판정 결과는 DB에 저장하고, 웹에서 조회하거나 보고서로 출력합니다.

## 웹 흐름

웹 기준 사용자 흐름은 다음과 같습니다.

1. 수집 결과 업로드 또는 수집 실행
2. 적용할 가이드라인 선택
3. 필요 시 회사 기준 PDF 업로드
4. 판정 실행
5. 결과 목록 확인
6. 항목별 상세 근거 확인
7. 보고서 다운로드

## 문서 유형별 청킹 전략

### 구조가 명확한 문서

예: 주통기, CIS Benchmark 일부 문서

- 항목 코드 중심 청킹
- `U-01`, `W-01` 같은 식별자를 활용
- exact lookup과 의미 검색을 함께 사용

### 자유 형식 문서

예: 회사 내부 기준서, 정책서, 표/문단 중심 PDF

- 제목, 소제목, 문단 단위 청킹
- 필요시 `overlap` 적용
- 페이지/섹션 메타데이터 강화

## DB 방향

현재는 즉시 확인 가능하도록 SQLite를 테스트용으로 사용하고 있습니다.

하지만 DB 방향은 아직 확정되지 않았습니다.

- 현재 목표: 구조를 유지한 채 DB 교체 가능하게 설계
- 테스트/시연: SQLite
- 향후 운영 후보: PostgreSQL 등

즉, 지금 중요한 것은 `DB 종류 확정`보다 `테이블 구조와 저장 흐름의 안정성`입니다.

## 배포 방향

예정 배포 방식은 다음과 같습니다.

- FastAPI 웹 애플리케이션
- 팀원 노트북을 서버로 사용
- 도메인 연결
- 브라우저에서 접근 가능한 형태 제공

운영 시 고려사항:

- HTTPS
- 포트 개방 및 방화벽
- DDNS 또는 고정 IP
- 장비 꺼짐/재부팅 리스크
- 백업 및 로그 관리

## 지금 프로젝트에서 중요한 설계 원칙

### 1. 수집과 판정을 분리한다

collector는 사실만 수집하고, 해석은 LLM이 담당합니다.

### 2. 가이드라인은 코드에 박지 않는다

가능한 한 가이드라인은 문서/DB/벡터 인덱스로 관리하고, 코드 수정 없이 추가 가능하게 합니다.

### 3. RAG는 선택된 기준 안에서 동작해야 한다

여러 가이드라인이 들어오면 검색 범위를 제한하지 않으면 의미가 흐려집니다.

### 4. 업로드 PDF도 기본 가이드라인과 같은 구조로 다룬다

기본 문서와 사용자 문서를 같은 메타데이터 구조로 넣어야 검색/설명/출처 추적이 쉬워집니다.

### 5. LLM 호출은 운영 제약을 전제로 설계한다

Gemini API 2.5 사용량 제한이 있으므로 큐, 재시도, 상태 저장이 필수입니다.

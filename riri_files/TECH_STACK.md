# 기술 스택 정리

## 핵심 방향

현재 기술 스택은 `바로 동작 확인 가능한 구성`과 `나중에 운영으로 확장 가능한 구성`을 함께 고려해서 잡고 있습니다.

즉, 지금은 테스트 가능한 단순성을 유지하되 아래 방향으로 확장하는 구조입니다.

- 웹 서비스화
- 다중 가이드라인 지원
- 업로드형 PDF 기반 RAG
- Gemini API 2.5 연동
- DB 교체 가능 구조

## 현재/예정 스택

| 영역 | 현재 또는 예정 기술 | 비고 |
|------|-------------------|------|
| 언어 | Python | 메인 구현 언어 |
| 웹 | FastAPI, Jinja2, Uvicorn | 브라우저 기반 서비스 |
| 수집 | subprocess, PowerShell, Linux 명령어 | OS 설정 정보 수집 |
| DB | SQLite 테스트 중, 운영 DB는 미정 | 교체 가능 구조 유지 |
| ORM/저장 계층 | SQLAlchemy | DB 종류 교체 대비 |
| 벡터 저장소 | Milvus | 가이드라인 검색 |
| 문서 처리 | PyMuPDF 기반, 향후 구조화 파서 확장 가능 | PDF 파싱 |
| 임베딩 | sentence-transformers 계열, 필요시 교체 | 한국어 문서 대응 |
| LLM | Gemini API 2.5 예정 | 현재 CLI 기반 흐름에서 전환 |
| 보고서 | HTML/PDF 출력 계층 | 결과 요약 및 배포 |

## LLM 스택

### 예정 방향

- `Gemini API 2.5` 사용
- 항목별 판정 호출
- rate limit 고려
- 실패 재시도 및 호출 상태 기록

### 운영 제약

현재 팀이 고려하는 제한:

- 분당 약 `15`회
- 하루 약 `1500`회

따라서 필요한 보조 구조:

- 요청 큐
- 동시성 제어
- 재시도
- 실패 로그
- 상태 저장

즉, LLM 모델 선택 자체보다 `호출 제어 계층`이 중요합니다.

## DB 스택

### 현재 상태

- 지금은 바로 확인하기 위해 `SQLite` 사용 가능
- 실제로 파일 기반 DB라서 개발/시연 단계에서 다루기 편함

### 중요한 점

SQLite를 최종 확정한 것은 아닙니다.

- 테스트용으로 빠르게 확인하기 좋음
- 추후 PostgreSQL 같은 운영 DB로 전환 가능성 있음
- 그래서 저장 로직은 DB 종류보다 `모델/리포지토리 구조`를 유지하는 쪽이 중요

## RAG 스택

### 현재 방향

- 가이드라인 PDF 파싱
- 청킹
- 임베딩
- 벡터 DB 저장
- 검색 후 LLM 판정

### 앞으로 바뀌는 핵심

- 가이드라인을 하나만 두지 않음
- `guideline_id` 중심으로 검색 범위 분리
- 업로드된 회사 PDF도 같은 구조로 저장

즉, 앞으로 RAG는 `문서 검색`이 아니라 `선택된 기준 집합 안에서의 검색`이 됩니다.

## 웹/배포 스택

### 예정 방식

- FastAPI 웹 서버
- 도메인 연결
- 팀원 노트북을 서버로 사용

### 같이 고려할 요소

- HTTPS
- 리버스 프록시
- 포트 개방
- 방화벽 설정
- DDNS 또는 고정 IP
- 서버 장비 꺼짐 리스크

개발용과 시연용은 가능하지만, 장기 운영이면 서버 안정성도 따로 고려해야 합니다.

## 문서 업로드 스택

회사별 기준 PDF를 다루기 위해 필요한 계층은 다음과 같습니다.

- 업로드 파일 저장
- PDF 파싱
- 청킹 전략 선택
- 메타데이터 생성
- 임베딩 생성
- 벡터 DB 적재

이때 문서마다 최소한 아래 메타데이터는 유지하는 것이 좋습니다.

- `guideline_id`
- `guideline_name`
- `source_type`
- `version`
- `page`
- `section_title`
- `chunk_id`

## 지금 기준 추천 정리

### 개발/테스트

- DB: SQLite
- 벡터 DB: Milvus
- 웹: FastAPI + Jinja2
- LLM: Gemini API 2.5 전환 준비

### 운영 확장 고려

- DB: PostgreSQL 등 검토 가능
- API 호출 큐 분리
- 문서 업로드/인덱싱 비동기화
- 웹 서버 외부 접근 구성

## 결론

현재 기술 스택의 핵심은 `하나의 완성된 최종 스택`을 확정하는 것보다, 다음 두 가지를 동시에 만족시키는 것입니다.

1. 지금 바로 테스트할 수 있어야 한다.
2. 나중에 운영 구조로 확장할 수 있어야 한다.

그래서 지금은 `SQLite + FastAPI + Milvus + Python` 조합이 현실적이고, 앞으로는 `Gemini API 2.5 + 다중 가이드라인 RAG + 웹 서비스 배포` 쪽으로 확장하는 흐름이 맞습니다.

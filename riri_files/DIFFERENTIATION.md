# 차별점 정리

## 한 줄 요약

이 프로젝트의 차별점은 `시스템 정보 수집기`가 아니라, `여러 기준 문서를 바꿔 끼울 수 있는 RAG + LLM 판정 플랫폼`이라는 점입니다.

## 기존 방식과의 차이

| 항목 | 기존 규칙 기반 점검 | 현재 프로젝트 방향 |
|------|---------------------|-------------------|
| 수집 스크립트 역할 | 수집 + 양호/취약 판단 | 수집만 수행 |
| 판정 방식 | 하드코딩된 if-else | RAG + LLM |
| 기준 문서 | 코드에 내장 | 문서 선택형 |
| 가이드라인 변경 대응 | 코드 수정 필요 | 문서 추가/교체 중심 |
| 회사별 기준 반영 | 어려움 | PDF 업로드로 반영 가능 |
| 설명 가능성 | 결과만 표시하기 쉬움 | 근거 문서 + 이유 설명 가능 |
| 서비스 형태 | 로컬 도구 중심 | 웹 기반 운영 가능 |

## 이 프로젝트가 강한 이유

### 1. 가이드라인을 바꿔도 구조를 유지할 수 있다

주통기만 지원하는 구조에서 끝나지 않고, 앞으로는 다음을 함께 다룰 수 있습니다.

- 주통기
- 다른 공공/민간 보안 가이드라인
- 회사 내부 기준 PDF

즉, 코드보다 `지식 베이스`가 중심이 되는 구조입니다.

### 2. collector가 중립적이다

수집 스크립트가 스스로 `양호`, `취약`을 말하지 않고 사실만 저장하면 다음 장점이 있습니다.

- 기준이 바뀌어도 재활용 가능
- 같은 수집 결과를 여러 가이드라인으로 다시 판단 가능
- LLM 판정 근거가 더 명확해짐

### 3. 사용자 선택 기반 RAG가 가능하다

앞으로는 웹에서 사용자가 기준을 선택하므로, RAG도 무조건 전체 문서를 뒤지는 것이 아니라 선택한 가이드라인 범위 안에서만 동작하게 됩니다.

이 점이 중요합니다.

- 의미 검색 품질 향상
- 다른 기준과의 충돌 감소
- 결과 해석이 쉬움

### 4. 업로드형 PDF 기준을 지원할 수 있다

회사별 운영 기준, 내부 보안 지침, 정책 문서 등은 공개 표준과 다를 수 있습니다.

이 프로젝트는 그 문서를 업로드해서 다음 흐름으로 바로 사용할 수 있게 가는 것이 특징입니다.

1. 문서 업로드
2. 문서 구조화
3. 임베딩 및 저장
4. 선택된 문서 기준으로 판정

### 5. 결과를 설명할 수 있다

단순히 `취약`이라고 끝나는 것이 아니라 아래를 함께 보여줄 수 있습니다.

- 어떤 기준 문서를 봤는지
- 어떤 부분이 근거였는지
- 왜 그렇게 판단했는지
- 어떤 조치가 필요한지

## 발표용 차별화 포인트

### 기술적 차별점

- 규칙 기반 스크립트가 아니라 RAG + LLM 기반 판정
- 다중 가이드라인 지원 구조
- 사용자 업로드 PDF 기반 커스텀 기준 지원
- guideline metadata 기반 검색 범위 제한

### 실무적 차별점

- 회사별 기준 반영 가능
- 기준 변경 시 코드 수정 부담 감소
- 결과 설명 가능
- 웹 서비스형으로 확장 가능

### 운영적 차별점

- Gemini API 2.5 호출 제한을 고려한 설계
- 테스트용 SQLite에서 운영용 DB로 이동 가능한 구조
- 팀 내부 장비 기반 시연/배포 가능

## 결론

이 프로젝트의 본질은 `자동 점검 스크립트`가 아니라, `문서를 기준으로 해석하고 설명하는 보안 판정 플랫폼`입니다.

그래서 가장 큰 차별점은 다음 한 문장으로 정리할 수 있습니다.

> 같은 수집 결과라도 어떤 기준 문서를 선택했는지에 따라 다시 판정할 수 있는 구조라는 점

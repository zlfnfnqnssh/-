## 필요한 역할 (4개)

### 역할 A: 시스템 정보 수집 모듈 개발

**한줄 요약:** 대상 시스템에서 보안 관련 정보를 자동 수집하는 스크립트 개발

**담당 모듈:** `collectors/`

**주요 업무:**
- Linux 수집 스크립트 개발 (Python + subprocess)
  - 계정 정보: `/etc/passwd`, `/etc/shadow`, `/etc/group` 파싱
  - 파일 권한: 주요 설정 파일 퍼미션 확인 (644, 600 등)
  - 서비스: `systemctl list-units`, 열린 포트 확인
  - 네트워크: `iptables`, `firewalld` 규칙 수집
  - 시스템 설정: `/etc/ssh/sshd_config`, PAM 설정 등
- Windows 수집 스크립트 개발 (Python + PowerShell)
  - 계정 정보: `net user`, 로컬 보안 정책 (`secedit`)
  - 레지스트리: 보안 관련 레지스트리 값 조회
  - 서비스: `Get-Service`, 시작 프로그램 목록
  - 방화벽: `Get-NetFirewallRule`
  - 감사 정책: `auditpol /get /category:*`
- 수집 결과를 통일된 JSON 스키마로 정규화

**산출물:**
- `collectors/linux_collector.py`
- `collectors/windows_collector.py`
- `collectors/normalizer.py`
- 수집 항목 정의서 (어떤 항목을 어떤 명령어로 수집하는지)

**필요 역량:** Python, Linux 명령어, PowerShell 기본, 시스템 관리 지식

**월별 계획:**
| 월 | 업무 |
|----|------|
| 3월 | 주통기 가이드라인 분석 → 수집 항목 목록 정의 |
| 4월 | Linux/Windows 수집 스크립트 개발 |
| 5월 | 수집 데이터 정규화, DB 저장 모듈과 연동 테스트 |
| 6월 | 다양한 환경 테스트, 버그 수정 |

- 검색 품질 검증 (수집 항목 → 올바른 가이드라인이 검색되는지)

**담당 모듈:** `database/`, `report/`

**주요 업무:**
- DB 스키마 설계 및 구축:
  - `scan_results` — 수집 결과 저장
  - `judgments` — LLM 판정 결과 저장
  - `reports` — 생성된 보고서 메타정보

- CRUD API 개발 (저장, 조회, 비교 함수)
- SQLite 기반 개발 → PostgreSQL 전환 고려
- 보고서 자동 생성 (PDF/HTML):

  - 이전 진단 대비 변화 추이
- 재진단 시 이전 결과 비교 기능

**산출물:**

- `report/templates/report_template.html`
- DB 설계 문서 (ERD)

**필요 역량:** Python, SQL, ORM(SQLAlchemy), HTML/CSS (보고서 템플릿), Jinja2

**월별 계획:**

### 역할 D: RAG 검색 & LLM 판정 파이프라인 + 프로젝트 총괄

**한줄 요약:** 수집 데이터와 가이드라인을 연결하여 AI가 취약점을 판정하는 핵심 파이프라인

**담당 모듈:** `engine/`

**주요 업무:**
- RAG 검색 시스템 구현:
  - 수집 항목을 임베딩 → VDB에서 관련 가이드라인 검색
  - 검색 쿼리 최적화 (item_name + collected_value 조합)
- LLM 판정 로직 개발:
  - 프롬프트 엔지니어링 (수집 데이터 + 가이드라인 → 양호/취약 판정)
  - 판정 결과 파싱 (JSON 출력 강제)
  - 재진단 시 이전 결과 포함한 프롬프트 설계
- 전체 파이프라인 오케스트레이션:
  - DB에서 항목 조회 → RAG 검색 → LLM 판정 → 결과 저장 → 반복
- 팀원 간 모듈 인터페이스 정의 및 통합 조율

**산출물:**
- `engine/rag_search.py`
- `engine/llm_judge.py`
- `engine/pipeline.py`
- 프롬프트 템플릿 문서

**필요 역량:** Python, LLM API 사용, RAG 개념, 프롬프트 엔지니어링, LangChain/LlamaIndex

**월별 계획:**

## 역할 간 협업 흐름

```
역할 A (수집) ──→ 역할 C (DB 저장) ──→ 역할 D (RAG+LLM 판정) ──→ 역할 C (리포트)
                                            ↑
                                       역할 B (VDB 검색)
```

| 연결 | 주고받는 것 |
|------|-----------|
| A → C | 수집 결과 JSON (정규화된 시스템 정보) |
| C → D | DB에서 수집 항목 조회 API |
| B → D | VDB 검색 인터페이스 (유사도 검색 결과) |
| D → C | LLM 판정 결과 JSON |
| C → 리포트 | 판정 결과 + 이전 비교 → 보고서 출력 |
